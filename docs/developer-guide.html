Altuwijri Ticket System

Developer Reference — v2.1 · Last updated 2026-06-28

Overview

The Altuwijri Ticket System (ATS) is an internal IT help-desk platform for the Altuwijri organisation. Employees submit IT support requests; technicians claim, work, and resolve them; managers run reports. It is a LAN-hosted full-stack web application accessible inside the corporate network.

Clean Architecture

The backend follows Clean Architecture (also known as Onion / Hexagonal). Dependencies flow strictly inward: outer layers depend on inner ones, never the reverse.

API Layer TaskFlow.Api
backend/src/TaskFlow.Api/
ASP.NET Core Web API. Controllers, middleware, DI registration, JWT config, SignalR hubs. Contains no business logic.
Application Layer TaskFlow.Application
backend/src/TaskFlow.Application/
All business logic. Services, DTOs, interfaces (IApplicationDbContext, ICurrentUser, IEmailService). Defines what the system does.
Infrastructure Layer TaskFlow.Infrastructure
backend/src/TaskFlow.Infrastructure/
Implements interfaces. EF Core DbContext, migrations, database seeder, email (SMTP), Identity setup.
Domain Layer TaskFlow.Domain
backend/src/TaskFlow.Domain/
Pure C# classes only. Entities (WorkTask, User, Branch, …) and enums (WorkTaskStatus, TaskPriority, …). Zero framework dependencies.

Soft delete: WorkTask has an IsDeleted flag. A global EF Core query filter in ApplicationDbContext automatically excludes deleted rows from every query:

modelBuilder.Entity<WorkTask>().HasQueryFilter(t => !t.IsDeleted);

Roles & Authorization

Roles are stored in ASP.NET Identity's AspNetRoles table. The Roles.cs file contains string constants.

HO-EmployeeRoles.HoEmployeeSame as Branch-Employee

Roles.EmployeeRoles is a helper array: [BranchEmployee, HoEmployee, CamEmployee].

Technician Ticket Visibility Rule

A technician sees on their board:

  • All tickets assigned to them (any project).
  • All tickets unassigned whose CategoryId matches one of their assigned categories — regardless of project.
  • If the technician has no category assignments, they see all unassigned tickets.

Deployment Guide

Overview Two IIS applications on the same server: the Angular SPA at the root, the .NET API at /api. ESET Endpoint Security manages firewall rules — port 80 must be allowed for LAN subnet.

Backend (IIS) Deploy

  1. Touch app_offline.htm in the API folder to gracefully stop IIS from serving requests while files are locked.
    New-Item -ItemType File "D:\Projetcs\Cloude\deploy\api\app_offline.htm" -Force
  2. Publish the .NET project:
    dotnet publish src/TaskFlow.Api/TaskFlow.Api.csproj -c Release -o D:\Projetcs\Cloude\deploy\api_pub
  3. Robocopy published output to IIS folder:
    robocopy D:\Projetcs\Cloude\deploy\api_pub D:\Projetcs\Cloude\deploy\api /MIR /NFL /NDL
    Robocopy exit codes 1–3 are success. Only ≥ 4 is an error.
  4. Remove the offline marker to bring the API back online:
    Remove-Item D:\Projetcs\Cloude\deploy\api\app_offline.htm -Force

Frontend (IIS) Deploy

  1. Add Node to path (if needed in PowerShell session):
    $env:Path += ";C:\Program Files\nodejs"
  2. Build Angular:
    cd frontend; node ".\node_modules\@angular\cli\bin\ng.js" build --configuration production
  3. Robocopy the browser output:
    robocopy frontend\dist\taskflow\browser D:\Projetcs\Cloude\deploy\web /MIR /NFL /NDL
ESET Firewall If inbound connections fail on port 80 after deploy, open ESET Endpoint Security → Network protection → Rules, and verify the rule allowing inbound TCP 80 for w3wp.exe is active.

Database Migrations

  1. Ensure connection string in appsettings.json is correct.
  2. Add migration: dotnet ef migrations add <Name> -p TaskFlow.Infrastructure -s TaskFlow.Api
  3. Apply: dotnet ef database update -p TaskFlow.Infrastructure -s TaskFlow.Api
  4. The seeder (DatabaseSeeder.cs) runs automatically on first start to seed roles and admin user.

Extending the App

Adding a new report endpoint

  1. DTO — add a new record to ReportDtos.cs
  2. Interface — add the method signature to IReportService in ReportService.cs
  3. Service — implement the method in the ReportService class (before // ----- helpers -----)
  4. Controller — add a [HttpGet("...")] action to ReportsController.cs
  5. Frontend model — add the TypeScript interface to models.ts
  6. Frontend service — add a method to ReportService in data.services.ts
  7. UI — add the mode to reports.ts mode union, add a segment button, load method, and template section

Adding a new ticket category (via code)

  1. In DatabaseSeeder.cs, add a new TicketCategory to the seed block.
  2. Or use the Admin → Ticket Categories UI at runtime — no code change needed.

Adding a new user role

  1. Add the role name string constant to Roles.cs
  2. Seed the role in DatabaseSeeder.cs
  3. Add [Authorize(Roles = "...")] to the relevant controllers
  4. Update role checks in shell.ts nav guards (auth.hasRole(...))
  5. Update roleGuard data arrays in app.routes.ts
Hot tip: hand-written mappings The project uses MappingExtensions.cs instead of AutoMapper. When you add a new property to a DTO, update the corresponding ToListItem() or ToDetail() call AND the DTO record constructor in the same file.
